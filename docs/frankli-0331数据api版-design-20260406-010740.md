## Problem Statement
No single mobile app combines live account-connected spending tracking, net worth visibility, and FIRE-specific projection in an emotionally engaging, mobile-first daily experience. Users are stuck between generic apps (Empower, YNAB) that ignore FIRE goals and complex planning tools (ProjectionLab) that feel like filing taxes rather than daily motivation.

## Demand Evidence
- Friends in the founder's circle have said they'd pay for it
- At least one person urgently asked when it ships
- Market context: 80-96% of personal finance app users quit within 1-3 months — the category is littered with abandoned apps, but no one has solved daily return motivation

## Status Quo
Target users currently manage FIRE tracking via manual spreadsheets (Excel/Notion). The workaround costs hours per month, is error-prone, has no live bank data, and provides no FIRE-specific metrics (distance to FIRE date, savings rate progress, scenario modeling).

## Target User & Narrowest Wedge
**Two user personas:**
1. FIRE beginners: confused by numbers, need clarity on "how far am I from financial independence" — higher acquisition potential, larger market
2. FIRE practitioners: already tracking manually in spreadsheets, want automation — higher retention, self-selecting motivation

**Narrowest wedge:** Any person who has heard of FIRE and googled "how much do I need to retire early" — they have the motivation but no daily tracking system. The minimum valuable product is: connect your accounts → see your FIRE date → watch it move.

## Constraints
- iOS SwiftUI, pure Apple frameworks, no third-party libraries
- Supabase backend with Plaid integration already built
- Dark theme, fire gradient design system established
- Single founder building

## Premises
1. Core value = integrating fragmented FIRE tools into one emotionally resonant app, NOT deeper calculation features (do not go the ProjectionLab complexity route)
2. FIRE beginners are the primary growth market (larger, lower acquisition cost) — though FIRE practitioners will have higher Day-60 retention
3. The biggest technical challenge (Plaid + Supabase data integration) is already solved. The bottleneck is now user retention
4. Differentiation: FIRE-native + emotionally engaging design, competing on product feel vs. Empower/YNAB's generic finance-app approach

## Approaches Considered
### Approach A: Emotional Countdown Minimal MVP
Skip Plaid, manual account entry, FIRE countdown clock + daily check-in. Ship in 2-4 weeks. Tests retention hypothesis directly. Discards already-built Plaid integration.

### Approach B: Full FIRE Dashboard (SELECTED)
Ship the current version with Plaid bank sync. FIRE countdown as hero metric. Basic FIRE calculation only at launch (not Coast/Barista/Fat FIRE variants — deferred to v2). Daily return mechanism via push notifications. 6-10 weeks to refine onboarding and trust flow.

Note on Approach A/B tension: Premise 3 says Plaid is solved and retention is the bottleneck — which would suggest Approach A (test retention first). Approach B is selected because the Plaid investment is real, but the retention concern from Premise 3 must be addressed inside Approach B, not by abandoning Plaid. The fix: solve retention WITHIN the current build, not by starting over.

### Approach C: Community + Product Dual Track
App + content account simultaneously. Consistent with long-term FIRE community vision. Not recommended for now — splits focus.

## Recommended Approach
**Approach B** — ship the current version, but solve two things before launch:
1. **Trust bridge** — the bank connection onboarding screen must explain Plaid's read-only model, no credential storage, and mention that Empower and other major apps use the same provider. Without this, most users will bounce at the connection step.
2. **Daily return hook** — add a morning push notification + visible progress delta on the home screen. The FIRE date should update visibly when spending or saving changes.

## MVP Feature Scope (Approach B — In Scope for v1)
**Core screens:**
- Onboarding: net worth + income input → FIRE date reveal (the "aha" moment)
- Home: FIRE countdown displayed as "X years, Y months" (not days — too abstract; not a calendar date — too distant). Net worth sparkline below. Savings rate badge.
- Cashflow: spending vs. income by month. Each spending category shows FIRE impact ("dining out cost you 3 days of FIRE")
- Investment: portfolio balance + allocation. Simple, not granular.
- Settings: account management, FIRE parameters (target retirement spend, current savings rate)

**Daily return mechanism:**
- Morning push notification: "Your FIRE date moved 2 days closer yesterday" or "You spent $47 on subscriptions yesterday — want to review?"
- Home screen shows delta from yesterday: "+2 days progress" or "-1 day (spending spike)"

**Trust bridge (onboarding):**
- Before Plaid link: one screen explaining read-only access, Plaid's role, which major apps use it
- "Read-only" badge visible permanently in account settings

**Out of scope for v1:**
- Coast FIRE / Barista FIRE / Fat FIRE scenario modeling (v2)
- Social sharing features (v2)
- Community features (v3)
- Android (post-iOS validation)

## Monetization (Default Assumption for v1)
Freemium: free for up to 2 connected accounts, $4.99/month or $39.99/year for unlimited accounts + FIRE scenario modeling (v2). Rationale: low enough to be impulse-buy for motivated FIRE users, high enough to fund development. Validate with first 100 users before committing.

## Success Criteria
**Pre-launch (TestFlight beta, first 20 users):**
- Bank account connection completion rate measured in TestFlight sessions
- Day-7 retention ≥ 30% target (if below 20%, solve trust/return hook before public launch)

**Post-launch:**
- Day-7 retention ≥ 30% (industry average is 10-15% for finance apps)
- Day-30 retention ≥ 15%
- Bank account connection rate > 40% of onboarded users (revised from 50% — first-time finance apps typically see 20-35%)
- At least 10 App Store reviews mentioning FIRE specifically in first month

## Dependencies
- **Plaid production credentials** — confirm approval status. If pending, this is a multi-week blocker that changes the timeline.
- Supabase Edge Functions deployed and stable
- Apple Developer account in good standing; budget 1-2 weeks for App Store review of financial data app (first submission often gets enhanced review)
- Push notification entitlements set up in Xcode project

## The Assignment
Before you adjust any more features: **put the current version in front of 5 strangers (not friends) and watch them try to connect their bank account without helping them.** Count how many make it through. That drop-off rate is your #1 product problem. Every feature decision should be subordinate to solving that funnel.

If Plaid production credentials are not yet approved: confirm this first. Everything else is irrelevant if the connection flow doesn't work.

## Reviewer Concerns (Unresolved — Engineering Spec Needed Before Build)
These are valid questions that need answers before engineering begins, but are outside the scope of a product strategy session:
1. **FIRE date formula** — define the calculation (25x annual expenses / 4% rule assumed, but needs confirmation). Input variables: current net worth, annual spending, expected return rate.
2. **Delta computation** — how does "FIRE date moved N days" work with Plaid's 24-48h data latency? Define stale data handling.
3. **Freemium paywall mechanics** — StoreKit integration, account limit enforcement, downgrade behavior. Move to v1 scope or explicitly defer to v2 before sprint planning.
4. **Push notification trigger** — which Supabase Edge Function fires at what time? Cron schedule, fan-out logic, APNs integration.
5. **Cashflow FIRE impact formula** — "N days of FIRE" per dollar of spending needs a clear derivation.
