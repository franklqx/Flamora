## Vision

### 10x Check
A user wakes up every morning and their lock screen notification says: "Your FIRE date got 2 days closer yesterday. Now it's 14 years 3 months away." They open the app and see — not an account balance — but a burning countdown framed as progress: "You're 28% of the way there." Every time they skip a restaurant meal, the countdown moves. They screenshot it and post to Reddit r/financialindependence. Someone in the comments asks "what app is this?"

That's Flamora. The first finance app where your FIRE date is the hero, not your account balance.

### Platonic Ideal
The ideal Flamora feels like a fitness tracker for your financial independence. Every day has a number. Every number moves. The movement feels earned. Users don't track because they have to — they track because watching the countdown drop is genuinely satisfying. The UI is dark, warm, alive. Fire gradient on the progress bar. Milestones celebrated. Social by design: FIRE milestone screenshots are something people want to share.

### 2. Notification System (2 tiers for v1)

**Daily (8am local):** Inspirational quote push
- Content: reuse existing `dailyQuotes` array in `JourneyView.swift`
- Server: Supabase Edge Function cron via `pg_cron` — schedule at **14:00 UTC** (covers 6 AM PT / 9 AM ET — acceptable for v1 North American user base)
- v2 upgrade path: store `utc_offset` in `profiles` table, bucket users into timezone groups, schedule multiple pg_cron jobs per bucket
- APNs delivery: Edge Function `send-fire-notification` reads all `apns_token` values from `profiles` table, sends per-user push via APNs HTTP/2 API
- APNs auth: JWT signing (ES256). Private key stored as Supabase secret `APNS_PRIVATE_KEY`. Team ID stored as `APNS_TEAM_ID`, Key ID as `APNS_KEY_ID`. Signing handled in Edge Function using Web Crypto API (available in Deno runtime — no external lib needed).

**Weekly (Sunday 18:00 UTC = ~10 AM PT / 1 PM ET):** FIRE progress summary
- Content: "This week your savings rate was Y%. Keep it up." (avoid "X days closer" — Plaid data latency is 24-48h, making daily deltas unreliable)
- Computation: server-side in `send-fire-notification` Edge Function — call `get-spending-summary` with `startDate = 7 days ago`, `endDate = today`. No new API field needed.
- Framing: savings rate % (computable from existing data) is more reliable than FIRE date delta for weekly cadence

**Monthly notification: DEFERRED to v2.** Daily + weekly is sufficient retention for v1. Monthly adds pg_cron complexity without proportional lift for a single-founder launch.

**Behavior:**
- "Enable Notifications" → `UNUserNotificationCenter.current().requestAuthorization(options: [.alert, .sound, .badge])`
- On grant → `UIApplication.shared.registerForRemoteNotifications()` → `application(_:didRegisterForRemoteNotificationsWithDeviceToken:)` → save token to `profiles.apns_token`
- "Not now" → skip silently, advance to next step. Do NOT re-prompt in-app. User can enable in iOS Settings.
- Anti-pattern avoided: NO 3-column feature grid, NO "Here's what you'll get:" list. Single value proposition.

- `UNUserNotificationCenter` permission request at END of onboarding (after `OB_RoadmapView` step 15, before app main flow)
- On permission grant: register for remote notifications → save APNs token to Supabase `profiles.apns_token`

**Supabase schema change:**
```sql

ALTER TABLE profiles ADD COLUMN IF NOT EXISTS apns_token TEXT;
ALTER TABLE profiles ADD COLUMN IF NOT EXISTS apns_token_updated_at TIMESTAMPTZ;
```
