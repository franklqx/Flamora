## Overall Verdict

**6/10 — NOT SHIP READY.** The core product works. The FIRE calculation is server-side and correct. Plaid is fully integrated. RevenueCat paywall exists. But three things are hard blockers: hardcoded test credentials that would fail in production, 66 `print()` statements leaking sensitive financial data to the console, and zero automated tests (any regression before launch goes undetected). ## Section 1: Architecture Review

### What it does well
- Clean separation: APIService handles network, PlaidManager owns bank connection state, SubscriptionManager owns subscription state. All as `@Observable` singletons injected via `.environment()`.
- Smart 401 handling: `APIService.perform<T>()` proactively refreshes tokens 60s before expiry, retries on 401. Covers 90% of real-world auth failures.
- `TabContentCache` prevents unnecessary re-fetches across tab switches. The cache invalidation on bank disconnect is clean.
- The `OB_ContainerView` step machine (18 steps, step 16 skipped/merged) is readable and navigable.

### Risks
1. **Dual Supabase clients.** `APIService` and `SupabaseManager` both hold separate Supabase clients initialized with the same hardcoded credentials. If the anon key ever needs rotating, it must be changed in two places. Merge into a single shared client.
2. **No session locking on concurrent 401.** If 3 API calls hit simultaneously and all get 401, all 3 will attempt `refreshSession()` concurrently. The Supabase SDK may handle this, but it's worth verifying — concurrent refresh with the same refresh token can invalidate it.
3. **`skipToMain = false` flag in ContentView.** A debug bypass that bypasses auth gating. Must be removed before production. A determined user could flip this.

### Action items
- [ ] Merge `APIService.supabaseURL`/`anonKey` and `SupabaseManager.client` into a single `SupabaseManager.client` used by both
- [ ] Add session refresh locking: `private var isRefreshing = false` + actor or AsyncLock to serialize refresh
- [ ] Remove `skipToMain` flag from ContentView entirely

## Section 2: Error & Rescue Map

### Current error handling coverage

| Flow | Error handled? | User sees? |
|------|---------------|------------|
| Network timeout (API call) | No timeout set | Spinner forever |
| 401 unauthorized | Yes — retry once | No |
| 403 premium required | Yes — shows paywall | Yes |
| Plaid link-token fetch fails | Logged, `isConnecting = false` | No |
| Plaid exchange fails | Logged, silent | No |
| Bank disconnect fails | Logged, silent | No |
| FIRE goal API fails | Falls back to nil state | Shows empty card |
| Spending summary fails | Falls back to `.empty` | Shows $0 |
| RevenueCat check fails | Defaults `isPremium = false` | No |
| Decode error (any API) | Logged | No |

**The pattern:** errors are logged but not surfaced. A user whose Plaid exchange fails has no idea. They think they're connected. They'll see $0 in their investment tab and blame the app.

### Required error surfaces
- **Toast/banner system** — a 3-second error banner at bottom of screen is sufficient. One component, used everywhere.
- **Retry affordance on data screens** — if `JourneyView.loadData()` fails, show "Couldn't load your data. Tap to retry." Not a full error screen, just an inline message.
- **Plaid failure alert** — if `exchangePublicToken()` fails, show: "Couldn't connect your account. Try again." with a retry button.

### Action items
- [ ] Add `request.timeoutInterval = 20` to all `URLRequest` objects in `APIService`
- [ ] Create `ErrorBanner.swift` in `Components/` — a simple `@State var errorMessage: String?` bottom toast
- [ ] Wire `exchangePublicToken()` failure → show alert in PlaidManager's presenting view
- [ ] Wire `loadData()` failure in JourneyView, CashflowView, InvestmentView → inline error state

## Section 3: Security & Threat Model

### Critical issues

**Hardcoded Supabase anon key in source.**
`APIService.swift:15-16` and `SupabaseManager.swift:22-23` contain the full Supabase URL and anon JWT. This is now in git history. Anyone with repo access has it. It's also in the app binary — extractable with `strings` or a reverse engineering tool in under 5 minutes.

This is _not_ a catastrophic breach if Supabase Row Level Security (RLS) is correctly configured. The anon key is meant to be public-ish. But it means you're relying 100% on RLS to be your security layer. If any RLS policy has a hole, anyone can read any user's financial data.

**What to verify:**
- Every Supabase function must check `auth.uid() = user_id` before returning data
- Test this: call `get-spending-summary` with a valid anon key but someone else's user ID — it should return 403 or empty

**66 print statements with financial data.**
```swift
print("🏦 [PlaidManager] ✅ Bank linked! institution=\(connectedInstitutionName ?? "?"), accounts=\(response.data.accounts_linked)")
print("[APIService] 🔒 Refreshing token: expiresAt=\(current.expiresAt)")
print("📊 [APIService] Response: \(String(data: data, encoding: .utf8) ?? "(nil)")")
```

The last one is the worst — it prints the entire API response body, which includes account balances, transaction amounts, and Plaid tokens, to the device console. Anyone with a USB cable and Console.app can read your users' bank data. **This must be removed before TestFlight.**

**Test RevenueCat API key.**
`test_CTvrBqscaqNCibGtSMxCUeXbmae` — the prefix `test_` is RevenueCat's convention for sandbox keys. In production, ALL purchases will either fail or be treated as sandbox transactions. Your subscription revenue will be zero.

### Action items
- [ ] Move Supabase URL + anon key to `Config.xcconfig` (gitignored)
- [ ] Move RevenueCat API key to `Config.xcconfig` (production key, not test)
- [ ] Audit all 66 `print()` calls — remove or replace with `#if DEBUG` guards
- [ ] Test Supabase RLS policies: attempt cross-user data access with anon key
- [ ] Verify no Plaid access tokens appear in logs (they do currently in PlaidManager)

## Section 4: Data Flow & Edge Cases

### Main happy path (works)
App start → auth check → MainTabView → parallel API fetches → FIRE data displayed. This path is solid.

### Edge cases that are broken or untested

**"First open after Plaid connection."**
When user connects bank for the first time, `PlaidManager.lastConnectionTime` updates, which triggers `JourneyView`'s `.onChange` to reload. But if `get-investment-holdings` returns empty (Plaid hasn't synced yet — can take 24-48h), `InvestmentView` shows a ghost state. The ghost state exists and looks good, but there's no "your data is syncing" message to set expectation.

**"Plaid link token expires mid-flow."**
Plaid link tokens expire after 30 minutes. If a user starts Plaid, gets distracted, and returns 31 minutes later, the token is expired. The `PlaidLinkHostViewController` will fail silently. User sees a blank screen or dismissal. No error message.

**"User selected 0 accounts in Plaid."**
The `selectedAccountIds` array passed to `exchangePublicToken()` could be empty. The backend call will succeed (Plaid accepts it), but no accounts will sync. User will see ghost state forever. Add client-side validation: require at least 1 account selected.

**"Network loss mid-onboarding."**
If `OB_LoadingView` (step 14 — the FIRE calculation) loses network, `createUserProfile()` fails silently. The loading spinner runs until... it doesn't stop. There's no timeout on the loading view. User is stuck forever. Add a 30s timeout with "Something went wrong, try again" state.

**"User taps back during Plaid link."**
`onDismiss` callback in `PlaidLinkPresenter` is unused. If user dismisses Plaid without connecting, `isConnecting` should reset to false. Check this is handled.

### Action items
- [ ] Add "Syncing your accounts — this can take a few minutes" message in InvestmentView ghost state
- [ ] Add link token expiry detection (timestamp the token creation, check before presenting)
- [ ] Validate `selectedAccountIds.count > 0` before calling `exchangePublicToken()`
- [ ] Add 30s timeout to `OB_LoadingView` with retry state
- [ ] Verify `isConnecting = false` on Plaid dismiss

- `CashflowDetailBuilders.swift` and `InvestmentAllocationBuilder.swift` isolate view-model transformation cleanly.

### Issues

**Force unwraps on URLs (4 instances):**
```swift
// APIService.swift:113

// APIService.swift:117
var request = URLRequest(url: urlComponents.url!)
```
If `baseURL` is ever empty or contains an invalid character, this crashes the app. The only reason this hasn't crashed is because `baseURL` is hardcoded to a valid URL. Move to xcconfig + guard against nil.

**`ContentView.skipToMain` debug flag:** remove it.

**`SimulatorView` uses `MockData.apiFireGoal`:**
```swift

SimulatorSettings.fromAPI(MockData.apiFireGoal)
```
The simulator is using mock data, not the real `getActiveFireGoal()` response. This means the simulator shows different numbers than JourneyView. Inconsistency will confuse users.

**66 print() statements in production path** — see Section 3.

### Action items
- [ ] Replace 4 force-unwrap URLs with guard + throw `.invalidURL`
- [ ] Wire `SimulatorView` to use real `fireGoal` from `getActiveFireGoal()` instead of mock
- [ ] Remove or `#if DEBUG`-wrap all 66 `print()` calls
- [ ] Delete `skipToMain` from ContentView

## Section 6: Test Review

**Zero tests.** The `/Flamora appTests/` directory doesn't exist.

This is expected for a solo founder MVP and doesn't block launch. But it means any change to `APIService`, `PlaidManager`, or `SubscriptionManager` before launch could introduce an undetected regression. The risk window is small but real.

### Minimum pre-launch test surface (1-2 hours to write)
Don't write tests for everything. Write them for the 3 places where a silent bug would be hardest to catch:

1. **`APIService.perform<T>()`** — test that 401 triggers one refresh + retry, not infinite loop
2. **`SubscriptionManager.checkStatus()`** — test that `isPremium` is false when entitlements empty, true when "Flamora Pro" active
3. **`InvestmentAllocationBuilder`** — unit test the asset type mapping (the bucketing from Plaid asset types to UI categories)

These 3 tests cover the highest-risk paths. Write them in `Flamora appTests/` using `@testable import Flamora_app`.

- [ ] Write test for `APIService` 401 retry behavior

## Section 7: Performance Review

### What's fine
- `TabContentCache` prevents re-fetching portfolio history on tab switch. The cache is per range (1w, 1m, 3m, ytd, all) — correct.
- The parallel `async let` pattern in `JourneyView.loadData()` fetches FIRE goal, net worth, spending, budget simultaneously. This is right.
- `PlaidLinkPresenter` correctly uses `UIViewController` lifecycle for Plaid — no SwiftUI-specific hacks.

### What's slow or risky

**`getTransactions(page: 1, limit: 50)` + no pagination UX:**
`AllTransactionsView` loads 50 transactions with a "Load more" button. If a user has 200 transactions and taps "Load more" 3 times rapidly, 3 concurrent API calls fire. No debouncing, no cancellation. The last response to arrive wins (the `transactions` array will be whatever the last successful call returned). Add a `isLoadingMore` guard.

**`calculateAvgSpending(months: Int)` — expensive call, no cache:**
Called from `BS_SpendingBreakdownView`. This call likely does aggregation server-side. No client-side cache. If user navigates back and forward in Budget Setup, it re-fires. Add result to `TabContentCache` or use `@State` to retain it within the view's lifecycle.

**`OB_LoadingView` spinner has no timeout** — mentioned in Section 4. Also a performance perception issue: if the server is slow, user thinks the app is frozen.

### Action items
- [ ] Add `isLoadingMore: Bool` guard in `AllTransactionsView` before firing next page
- [ ] Cache `calculateAvgSpending()` result for the Budget Setup session
- [ ] Add 30s timeout to `OB_LoadingView`

## Section 8: Observability & Debuggability Review

**66 print() statements** — this is actually your entire observability story right now. When something breaks in production, you have zero insight because:
1. Users don't have Console.app open
2. You can't read their device logs
3. There's no remote logging

For a finance app handling bank connections, this is a real operational risk. When a user tweets "my bank won't connect," you'll have no server-side traces to debug.

### Minimum observability for launch
- **Server-side Plaid error logging:** the `exchange-public-token` Edge Function should log errors to Supabase's built-in function logs (they appear in the Supabase dashboard under Functions > Logs). Verify this works before launch.
- **Client-side: Supabase Analytics or a free Sentry free tier.** Sentry for iOS captures crashes and errors with stack traces. Free tier covers 5k errors/month. This catches force-unwrap crashes that you'd otherwise never know about.
- **RevenueCat dashboard** — already built-in once you switch to production key. Gives you purchase success/failure rates.

### What to remove
All 66 `print()` calls. Replace with `#if DEBUG` guards for the ones you want to keep during development:
```swift
#if DEBUG
print("🏦 [PlaidManager] Bank linked: \(institution)")
#endif
```

### Action items
- [ ] Wrap all print() with `#if DEBUG` before TestFlight
- [ ] Verify Supabase Edge Function logs are capturing errors (not just successes)
- [ ] Integrate Sentry free tier for crash reporting (add to approved CLAUDE.md exceptions)
- [ ] Set up RevenueCat production dashboard alerts for failed purchases

## Section 10: Long-Term Trajectory Review

### The bet you're making
You're building a daily-return app in the category with the worst Day-30 retention in mobile (10-15% for finance). The bet: FIRE-motivated users are different — they have intrinsic motivation that generic finance app users don't. The design doc's subagent raised this. You disagreed. The disagreement is defensible (FIRE practitioners self-select for high motivation) but it means your Day-7 retention benchmark matters more than it would for any other category.

**If Day-7 retention ≥ 20%, the bet is paying off. If it's < 15%, you have a retention problem, not a feature problem.** Don't add features to fix a retention problem — add push notifications and a daily hook.

### Where the trajectory goes

The v1 launch arc is correct:
1. Ship with Plaid sync + FIRE countdown hero
2. Add push notifications for daily engagement
3. Measure retention → optimize onboarding funnel
4. v2: FIRE scenario modeling (Coast, Barista, Fat) + FIRE impact per spending category
5. v3: Community / social sharing

The key decision point will come at ~1,000 users: is this a habit (daily opens) or a dashboard (monthly opens)? Daily habit = monetization ceiling is high. Monthly dashboard = you're competing with Empower and losing.

**Push notifications are the single highest-leverage item in the v1 plan.** Not because of the notification content itself — because the act of responding to a notification establishes a behavior loop. Build this before launch, not after.

### The moat
The FIRE calculation + Plaid data combination is not technically hard to replicate. Your moat is two things:
1. **Brand in the FIRE community.** The first app that gets recommended on r/financialindependence owns that community. Be first.
2. **Emotional design.** The fire gradient, the countdown, the "you're 28% there" framing — Empower won't copy this because it's not how big companies think. You can go faster on feel than they can.

### Action items
- [ ] Set Day-7 retention ≥ 20% as the explicit go/no-go gate before wider distribution
- [ ] Post to r/financialindependence in week 1 of TestFlight — not to acquire users, to get feedback
- [ ] Build push notifications before launch (in CEO plan, already scoped)
- [ ] After 100 users: run qualitative interviews with top 10% most active users — understand why they return

## Section 11: Design & UX Review

### What's working
The fire gradient design system is visually distinctive and consistent. The dark theme with fire-orange/purple gradient reads as premium. The card-based layout in JourneyView is scannable.

The onboarding has a proper "aha moment" at step 15 (`OB_RoadmapView`) — this is where the user sees their FIRE date for the first time. That's good product thinking.

### The FIRE countdown isn't the hero yet

Right now, `JourneyView` leads with `PortfolioCard` (net worth + investment balance). The FIRE countdown appears... somewhere below. This is the single most important UX fix before launch.

The user's first question when they open the app should be answered by the first card they see: "How close am I to financial independence?" Net worth is a means to an end. FIRE progress is the end.

**Fix:** Move FIRE countdown card to position 1. Display: `"You're 28% of the way there · 14 years 3 months to go"` with a progress bar. This is already in the CEO plan.

### The "14 years" problem

You raised this — long timelines feel overwhelming. The fix is the framing:
- ❌ "14 years to go" — feels like punishment
- ✅ "28% of the way there" — feels like progress
- ✅ "14 years, 3 months to go" — acceptable as secondary text if primary is the %

The percentage must come first, always. The countdown is confirmation of progress, not the headline.

### Missing screens
- **Trust bridge** (before Plaid) — already in CEO plan
- **Empty/loading states** — several cards show $0 or nothing when API is loading. Add a shimmer/skeleton state. The difference between "loading" and "error" is critical for user trust.
- **Notification permission request** — needs to be friendly and explain the value. "We'll tell you when your FIRE date moves" is the reason to grant permission.

### Action items
- [ ] Move FIRE countdown card to position 1 in JourneyView
- [ ] Display format: progress % first, years+months second
- [ ] Add skeleton/shimmer loading state to PortfolioCard, FIRECard
- [ ] Design trust bridge screen (in CEO plan — now needs implementation)
- [ ] Design notification permission request screen (end of onboarding, step 18)
