## P0 Bug 修复进展（关键阻塞）

### Bug 描述
**Plaid Link 连接成功后，返回主界面，所有 Tab 页面的按钮无法点击（触摸事件被拦截）。**

### 已尝试方案（全部失败或部分有效）

| 方案 | 结果 |
|------|------|
| 修改 `fullScreenCover` dismiss 逻辑（pendingLinkToken 时机） | 失败 |
| `@State showPlaidLink` + `capturedLinkToken` 独立变量 | 导致白屏 |
| `fullScreenCover(item:)` + `PlaidLinkItem` Identifiable wrapper | 失败 |
| `PlaidLinkRepresentable` 中先 dismiss linkVC 再调回调 | 失败 |
| `findLinkViewController()` 遍历 parent chain | 失败 |
| **UIWindow overlay（当前方案）** | 部分有效，见下 |

### 当前方案：UIWindow Overlay（`PlaidLinkPresenter.swift`）

**原理：** 不使用 SwiftUI `fullScreenCover`，改用独立 `UIWindow`（`windowLevel = .alert + 1`）呈现 Plaid Link，连接完成后销毁该 window。

**本次改动文件：**
- **新建：** `Flamora/Services/PlaidLinkPresenter.swift` — UIWindow overlay 核心逻辑
- **修改：** `Flamora/Services/PlaidManager.swift` — 移除 `pendingLinkToken`，直接调用 `PlaidLinkPresenter.shared.present(...)`
- **修改：** `Flamora/Flamora app/ContentView.swift` — 移除 Plaid fullScreenCover 相关代码，移除 `@Environment(PlaidManager.self)`

**可删除文件（已无引用）：**
- `Flamora/View/Shared/PlaidLinkRepresentable.swift` — 旧的 `UIViewControllerRepresentable` 方案，现已废弃

### 诊断发现的真实根因

通过在 `destroyWindow()` 中打印所有 window 信息，发现：

```
▸ UIWindow              level=0     hidden=false  interaction=true   key=true   ← 主 App window ✅
▸ UIWindow              level=2001  hidden=true   interaction=false  key=false  ← 我们的 overlay（已隐藏）✅
▸ UITextEffectsWindow   level=2002  hidden=false  interaction=true   key=false  ← 真正的拦截者 ❌
```

**根本原因：**
Plaid Link 内部使用 WKWebView 渲染界面。WKWebView 在 iOS 中使用 `UITextEffectsWindow`（level 2002）处理文字选中、放大镜等效果。当 WKWebView 被 dismiss 时，UIKit **没有自动隐藏**该 window，导致其以 `hidden=false + interaction=true` 状态停留在最顶层，拦截所有触摸事件。**这是 Plaid SDK 的清理疏漏，与我们用 fullScreenCover 还是 UIWindow 无关。**

### 当前状态

`destroyWindow()` 中已加入如下修复：

```swift
for w in scene.windows where w !== win && !w.isHidden && w.windowLevel > .normal {
    print("🔗 Hiding elevated window: \(type(of: w)) level=\(Int(w.windowLevel.rawValue))")
    w.isHidden = true
}
```

但测试结果显示 tab **仍然无法点击**，说明：
1. `UITextEffectsWindow` 被隐藏后，可能还有其他 window 或 view 在拦截触摸
2. 或者 `UITextEffectsWindow` 的隐藏没有生效（系统可能覆盖了我们的操作）

## 下一步排查方向（P0 继续）

### 方向一：验证 UITextEffectsWindow 是否真的被隐藏

在 `destroyWindow()` 结束后，增加延迟打印，确认 `UITextEffectsWindow` 的状态：

```swift
DispatchQueue.main.asyncAfter(deadline: .now() + 0.5) {
    for w in UIApplication.shared.connectedScenes
        .compactMap({ $0 as? UIWindowScene })
        .flatMap({ $0.windows }) {
        print("  [0.5s later] \(type(of: w)) hidden=\(w.isHidden) interaction=\(w.isUserInteractionEnabled)")
    }
}
```

### 方向二：遍历 view 层级查找拦截者

触摸拦截不一定来自 window，也可能是某个 view 的 `isUserInteractionEnabled`、`frame` 覆盖、或 `gestureRecognizer` 导致。

可用 `Xcode → Debug → View Hierarchy`（或 LLDB `po [UIWindow keyWindow] recursiveDescription`）在连接成功后截图 view 层级，查找覆盖整个屏幕的透明 view。

### 方向三：检查 WKWebView first responder 状态

Plaid WKWebView dismiss 时可能没有 resign first responder。尝试在 `tearDown()` 开始时：

```swift
UIApplication.shared.sendAction(#selector(UIResponder.resignFirstResponder), to: nil, from: nil, for: nil)
```

### 方向四：考虑完全不同的 Plaid 呈现方式

使用 `SFSafariViewController` 方式呈现 Plaid（如果 Plaid 支持 OAuth 模式），避免 WKWebView 的副作用。
